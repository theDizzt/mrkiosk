# 시스템 아키텍처 문서 초안
- 2026년 4월 8일 작성

## 1. 문서 개요
### 1.1 문서 목적
본 문서는 시니어 전용 고정형 MR 스테이션 기반 Text-Free 주문 보조 시스템의 전체 구조를 정의하고, 각 개발 파트가 공통 기준에 따라 구현 및 통합될 수 있도록 시스템 구성, 데이터 흐름, 좌표계 정합 방식, 상태 전이 구조, 모듈 간 인터페이스를 명세하는 것을 목적으로 한다. 본 프로젝트는 텍스트 기반 정보 해석이 어려운 시니어 사용자가 음성 안내와 혼합현실 시각 가이드를 통해 독립적으로 키오스크 주문을 수행할 수 있도록 설계된다 

### 1.2 시스템 목표
본 시스템의 목표는 사용자가 글자를 읽지 않고도 음성과 시각 가이드만으로 주문을 완료할 수 있도록 지원하는 것이다. 이를 위해 고정형 MR 스테이션 구조를 사용하여 좌표 드리프트를 최소화하고, 실제 버튼 위치와 가상 오버레이를 정합하며, 로컬 상태 전이 기반으로 빠르고 안정적인 인터랙션을 제공한다 

### 1.3 문서 범위
본 문서는 다음 내용을 포함한다.
- 전체 시스템 구성 요소와 역할
- 시스템 논리 구조와 데이터 흐름
- 시각 상태 동기화 구조
- 좌표계 및 공간 정합 기준
- 상태 전이(FSM) 구조
- 예외 처리 및 안전장치 개념
- 초기 개발환경 및 5~6주차 아키텍처 산출물 기준

## 2. 시스템 개요
### 2.1 시스템 컨셉
본 시스템은 Meta Quest 3 기반 MR 디바이스, 태블릿 기반 키오스크 시뮬레이터, ArUco 마커 인식 기반 로컬 비전 엔진, 음성 입력 처리 모듈, LLM 기반 예외 대응 모듈로 구성된다. 핵심 제어는 네트워크 중심 구조가 아니라 로컬 확정적 제어(Local Deterministic Control) 를 사용하는 FSM 기반 엔진이 담당하며, LLM은 주문 메인 흐름이 아니라 오조작 또는 모호한 발화 발생 시에만 보조적으로 개입한다 

### 2.2 핵심 설계 원칙
- Text-Free 인터페이스: 사용자에게 텍스트 판독을 요구하지 않고, 음성과 3D 시각 가이드를 중심으로 인터랙션을 제공한다 
- 고정형 스테이션 기반 안정성: 사용자의 위치와 기기 위치를 고정하여 MR 좌표 드리프트 문제를 완화한다 
- 비통신 상태 동기화: 별도 네트워크 통신 없이 태블릿 화면 내 마커 ID 변화를 통해 현재 상태를 동기화한다 
- 실시간성 중심 구조: 로컬 비전 엔진이 0.1초 수준의 빠른 피드백을 제공하도록 설계한다 

## 3. 전체 시스템 구성
### 3.1 구성 요소
시스템은 다음 5개 모듈로 구성된다.

#### 3.1.1 MR 클라이언트
Meta Quest 3 기반 장치에서 동작하며, 실제 키오스크 화면 위에 3D 가이드(글로우 링, 화살표 등)를 렌더링한다. Spatial Anchor 및 Passthrough 기능을 활용하여 실제 공간과 가상 오버레이를 정합한다 

#### 3.1.2 비전 엔진
OpenCV 기반 ArUco 마커 추적 및 6-DOF 포즈 추정을 수행한다. 태블릿 화면의 마커를 인식하여 현재 상태를 판별하고, 마커 기준 좌표계를 이용해 가이드 오브젝트의 위치를 계산한다 

#### 3.1.3 상태 제어 엔진
FSM 기반으로 동작하며 주문 단계, 오류 상태, 복구 경로를 관리한다. 정상 흐름의 주 제어를 담당하는 핵심 엔진이다 

#### 3.1.4 키오스크 시뮬레이터
태블릿 환경에서 동작하는 전용 UI 앱이다. 메뉴 선택 및 결제 흐름을 구현하고, 화면 내 마커 ID를 상태에 맞게 동적으로 변경하여 MR 측으로 시각적 상태 신호를 전달한다 

#### 3.1.5 음성 및 AI 보조 모듈
사용자의 발화를 STT로 텍스트화하고, 간단한 의도 분류 또는 예외 상황 판단에 활용한다. 메인 상태 진행은 FSM이 담당하며, LLM은 사용자가 당황하거나 복구가 필요한 상황에서만 안전장치로 개입한다 

## 4. 논리 아키텍처
### 4.1 상위 구조
아래 구조를 문서 그림으로 넣으면 좋다.
```
[User]
  ↓ voice / touch
[STT / Intent]
  ↓
[FSM State Controller] ←→ [LLM Fail-safe]
  ↓
[MR Guide Renderer]
  ↑
[Vision Engine: ArUco Detection + Pose Estimation]
  ↑
[Kiosk Simulator: UI + Dynamic Marker ID]
```

### 4.2 동작 흐름
- 사용자가 발판 위에 서고 시스템이 음성 안내를 시작한다.
- 사용자의 발화를 수신하여 초기 의도를 분석한다.
- FSM이 현재 주문 단계에 대응하는 상태를 활성화한다.
- 태블릿 화면에는 현재 상태에 대응하는 마커 ID가 표시된다.
- 비전 엔진이 마커 ID와 포즈를 인식한다.
- MR 렌더러가 해당 단계의 버튼 위치에 주황색 가이드를 투사한다.
- 사용자가 올바른 버튼을 누르면 시뮬레이터 상태가 바뀌고, 마커 ID도 다음 상태로 변경된다.
- 비전 엔진이 새로운 ID를 감지하면 FSM이 다음 상태로 전이된다.
- 오조작이나 장시간 정지가 발생하면 LLM 보조 모듈이 복구 안내를 제공한다. 이 흐름은 제안서 시나리오의 자연어 주문, 시각 안내, 물리적 상호작용, 위기 상황 복구, 단계별 내비게이션 구조와 대응한다 

## 5. 좌표계 및 공간 정합 구조
### 5.1 좌표계 정의
본 시스템은 최소 3개의 좌표계를 사용한다.

#### 5.1.1 Marker Coordinate
ArUco 마커를 기준으로 생성되는 좌표계다. 비전 엔진의 기준 좌표계이며, 포즈 추정 결과의 직접 출력 기준이 된다.

#### 5.1.2 Kiosk UI Coordinate
태블릿 화면 내 버튼의 2D 위치를 표현하는 좌표계다. 메뉴 버튼, 결제 버튼, 확인 버튼 등의 위치가 이 기준으로 정의된다.

#### 5.1.3 MR World Coordinate
Quest 3 내 실제 공간 좌표계다. 최종적으로 3D 가이드 오브젝트가 배치되는 좌표계다.

## 5.2 좌표 변환 개념
시스템은 다음 변환 단계를 사용한다.

UI Button Position
→ Marker-relative Offset
→ Marker Pose (rvec, tvec)
→ MR World Position
→ 3D Guide Render

즉, 버튼은 UI 상의 절대 픽셀 좌표로만 쓰이지 않고, 마커 기준 상대 오프셋 데이터로 저장된 뒤, 실시간 포즈 추정값을 이용해 MR 월드 좌표로 변환된다. 제안서에서도 마커 기준 고정 오프셋 데이터를 적용한 3D 가이드 렌더링과 좌표계 동기화 로직 구현이 핵심 과업으로 명시돼 있다 

## 5.3 정합 목표
- 실제 버튼과 MR 오버레이 간 오차 1cm 이내 유지 
- 마커 인식부터 가이드 갱신까지 0.1초 이내 지연 목표 
- 네트워크 단절 시에도 로컬 마커 인식 기반 동작 유지 


## 6. 상태 전이 아키텍처
### 6.1 상태 제어 원칙
시스템의 메인 제어는 FSM 기반으로 수행한다. 상태는 사용자 주문 흐름에 맞춰 정의되며, 각 상태는 하나 이상의 마커 ID와 대응한다. 상태 전이는 사용자의 음성 입력, 시뮬레이터 UI 이벤트, 마커 ID 변화, 오류 발생 여부를 기준으로 이루어진다 

### 6.2 초기 상태 예시
초기 문서 초안에서는 아래처럼 잡으면 충분하다.
```
상태명	설명	입력 조건	출력
IDLE	사용자 대기	사용자 없음	대기 안내
LISTENING	발화 수신	사용자 감지	STT 시작
MENU_GUIDE	메뉴 선택 유도	메뉴 의도 인식	메뉴 버튼 하이라이트
OPTION_GUIDE	옵션 선택 유도	메뉴 선택 완료	옵션 버튼 하이라이트
PAYMENT_GUIDE	결제 유도	옵션 완료	결제 버튼 하이라이트
CONFIRM	주문 완료 확인	결제 완료	완료 피드백
ERROR_RECOVERY	오조작 복구	예상 외 상태 감지	복구 안내
FAIL_SAFE	장기 정지/혼란 대응	3초 이상 지연, 혼란 발화	LLM 음성 보조
```

### 6.3 상태 전이 규칙 예시
```
IDLE → LISTENING → MENU_GUIDE → OPTION_GUIDE → PAYMENT_GUIDE → CONFIRM
                      ↘
                    ERROR_RECOVERY → MENU_GUIDE 또는 OPTION_GUIDE
                                 ↘
                                FAIL_SAFE
```

### 6.4 마커 ID 설계 원칙
문서 단계에서는 아직 ID 숫자를 최종 고정하지 않아도 되지만, 원칙은 먼저 정해야 한다.
```
예:
0~9: 시스템 제어 상태
10~19: 메뉴 선택 상태
20~29: 옵션 상태
30~39: 결제 상태
90대: 오류/복구 상태
```
이렇게 해두면 나중에 상태 테이블과 디버깅 로그가 훨씬 깔끔해진다.

## 8. 예외 처리 구조
### 8.1 정상 흐름과 예외 흐름 분리
정상 흐름은 FSM이 담당하고, 비정상 흐름만 LLM이 담당한다. 이는 제안서의 “LLM의 역할을 주문 프로세스 외부의 모호한 질문이나 오조작 발생 시에만 한정”한다는 설계 원칙과 일치한다 

### 8.2 예외 유형
- 잘못된 버튼 클릭
- 광고/초기화면 등 예상 외 화면 전환
- 마커 미검출 또는 잘못된 ID 검출
- 사용자 3초 이상 정지
- 혼란을 나타내는 발화 발생

### 8.3 대응 방식
우선순위 1: 상태 마커 기반 즉각 복구
우선순위 2: FSM 복귀 경로 안내
우선순위 3: LLM 기반 정서적 안정 및 상세 안내 제공

이 구조는 제안서 시나리오의 위기 상황 대응 1, 대응 2와 직접 연결된다.

5주차

## 11. 결론
본 시스템 아키텍처는 단순한 MR 시각화 구조가 아니라, 고정형 스테이션 환경을 기반으로 한 좌표 정합, 비통신 상태 동기화, FSM 중심 로컬 제어, 예외 상황에서의 LLM 보조 개입을 결합한 복합 구조다. 따라서 초기 5~6주차의 핵심은 구현보다 먼저 구조를 명확히 정의하고, 좌표계와 상태 전이, 인터페이스 규격을 팀 공통 기준으로 문서화하는 것에 있다. 이는 이후 통합 테스트와 접근성 검증의 기반이 된다
